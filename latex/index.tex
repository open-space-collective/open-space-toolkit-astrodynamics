By default, O\+S\+Tk fetches the ephemeris from J\+PL, Earth Orientation Parameters (E\+OP) and leap second count from I\+E\+RS.

As a result, when running O\+S\+Tk for the first time, it may take a minute to fetch all the necessary data.

O\+S\+Tk has a cross-\/validation framework built into it (inside the {\ttfamily /validation} folder), which allows us to compare its end-\/to-\/end propagation accuracy of a full \char`\"{}mission sequence\char`\"{} to other flight qualified tools like G\+M\+AT and Orekit.\hypertarget{index_autotoc_md18}{}\doxysubsubsection{Scenario Definition}\label{index_autotoc_md18}
A standardized input format (a yaml file) to define the particular mission sequence scenario is used, and is tooling agnostic. An example of it is shown below.


\begin{DoxyItemize}
\item The spacecraft\textquotesingle{}s parameters and initial state are defined under the {\ttfamily spacecraft} header
\item The sequence header contains the dynamic forces to be applied during propagation, the type of numerical propagator to use, and the segments (burning or coasting) to execute during the mission sequence
\item Finally, the output header contains the quantities that we want to output (in this case time elapsed, position, and velocity), and the interval at which to report them
\end{DoxyItemize}

\hypertarget{index_autotoc_md19}{}\doxysubsubsection{Cross Validation and Accuracy Tolerances}\label{index_autotoc_md19}
The validation framework takes this yaml scenario definition (from the {\ttfamily /validation/data/scenarios} folder) and runs the scenario using O\+S\+Tk. Next, it compares the specified outputs (quantities like the position, velocity, mass, acceleration of the spacecraft) at the desired reporting step generated by O\+S\+Tk with each external validation tool (whose outputs were generated via the same process and are in the {\ttfamily /validation/data/gmat\+\_\+astrodynamics} or {\ttfamily /validation/data/orekit\+\_\+astrodynamics} folder as {\ttfamily .csv}), to ensure they are within a certain tolerance of each other.

{\bfseries{To add a scenario\+:}}
\begin{DoxyEnumerate}
\item Write a scenario definition using the {\ttfamily .yaml} format above, and name it {\ttfamily scenario\+X-\/mission-\/sequence.\+yaml}, and put it in wth the others. If the format isn\textquotesingle{}t correct or there is another issue, O\+S\+Tk will throw an error when validating.
\item Generate the {\ttfamily .csv} outputs from the external tools you want to compare O\+S\+Tk against (repos with an G\+M\+AT and Orekit wrapper that can ingest this {\ttfamily .yaml} format do exist and will be open sourced soon).
\item Add another test case to the {\ttfamily Framework.\+validation.\+cpp} file, with the scenario name, the tool to compare against, and the output quantities and comparison tolerances for those. An exampe of this is provided below.
\end{DoxyEnumerate}
